# Y2038 Preservation Workflow

## What is the Year-2038 Problem?

Most 32-bit operating systems and toolchains represent time as a signed
32-bit integer counting seconds since 1970-01-01 00:00:00 UTC (`time_t`).
This counter will overflow on **2038-01-19 03:14:07 UTC**, wrapping back to
a large negative value and causing a range of failures: incorrect timestamps,
log corruption, certificate/key expiry, filesystem inconsistencies, and
application crashes.

Systems and software that were built before this deadline—and are no longer
actively maintained—are at the highest risk.  Legacy 32-bit architectures
(i386, ARMv4–v6, MIPS-o32, …) are the primary concern because:

1. Newer 64-bit platforms already use 64-bit `time_t`.
2. Many 32-bit codebases are frozen and will not receive upstream fixes.
3. Existing binaries on such systems may be compiled with the old 32-bit ABI
   and cannot simply be recompiled.

## Why store patches/diffs instead of rewriting history?

**Immutable baselines + forward-applied patches** is the correct preservation
mechanic for a museum archive:

- The original source snapshot is unchanged and still verifiable by hash.
- Each fix is a discrete, reviewable, auditable diff.
- A patchset bundles multiple diffs into an ordered series that can be applied
  reproducibly to any copy of the baseline.
- Future researchers can see exactly what changed, why, and in what order—
  without needing to trust a rewritten or force-pushed history.

## Recommended workflow

```
baseline artifact  ──►  patchset  ──►  (optional) derived artifact
       │                  │                       │
   objects/          patchsets/y2038/        artifacts/
   artifacts/        patches/
```

### Step 1 – Import patch files

```
legacyai patch add path/to/fix-time_t.patch --tag y2038 --name "Fix time_t overflow in libc"
```

Each patch is stored in `patches/<sha256>.patch` alongside a YAML metadata
record `patches/<sha256>.yml`.  The SHA-256 hash of the file is its permanent
identifier.

### Step 2 – Create a patchset

```
legacyai patchset create y2038:fix-libc-time_t \
    --description "Backport 64-bit time_t to musl libc 1.1.x" \
    --baseline libc-1.1.24-src \
    --source-hash <expected sha256 of baseline tarball>
```

This creates `patchsets/y2038/fix-libc-time_t.yml`.

### Step 3 – Add patches to the patchset in order

```
legacyai patchset add y2038:fix-libc-time_t \
    --patch <sha256-prefix-or-full> \
    --order 1 \
    --note "Widen time_t typedef to long long"

legacyai patchset add y2038:fix-libc-time_t \
    --patch <sha256-of-second-patch> \
    --order 2 \
    --note "Update syscall wrappers to use __time64_t"
```

### Step 4 – Create a binding

```
legacyai bind source-patch \
    --source libc-1.1.24-src \
    --tag y2038 \
    --patchset y2038:fix-libc-time_t
```

This creates a record in `bindings/source-patch/<uuid>.yml` that links the
baseline, the patchset, and any tags together.

### Step 5 – Verify integrity

```
legacyai verify
```

`verify` checks:
- Every patch file on disk matches its stored hash.
- Every patchset references patches that exist in `patches/`.
- Every binding references patchsets and source artifacts that exist.
- Baseline and output artifact records exist where referenced.

Exit code is `0` when clean, `1` when errors are found.

## Key dates

| Date | Event |
|------|-------|
| 2038-01-19 03:14:07 UTC | 32-bit `time_t` overflows |
| 2038-01-19 onwards | Systems without patches begin exhibiting failures |
| ~2040–2042 | Cascading failures in long-running daemons, certificate stores, build tools |

## Directory layout

```
Legacy64/
├── patches/                   Raw patch files + metadata
│   ├── <sha256>.patch
│   └── <sha256>.yml
├── patchsets/
│   └── y2038/                 Y2038 patchset metadata
│       └── <name>.yml
├── bindings/
│   └── source-patch/          Source→patch bindings
│       └── <uuid>.yml
├── artifacts/                 Artifact metadata records
│   └── <id>.yml
├── objects/                   Content-addressed binary store
│   └── ...
└── schemas/                   JSON Schema definitions
    ├── patch.schema.json
    ├── patchset.schema.json
    └── binding.schema.json
```
